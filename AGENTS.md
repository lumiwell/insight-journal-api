# Journal API - Backend Agent Rules

You are working on the `journal-api` backend project. Always adhere to the following architectural invariants and tech stack choices:

## Tech Stack
- **Framework**: FastAPI (Async)
- **Database**: PostgreSQL 16 (with `pgvector` enabled for future RAG).
- **ORM**: SQLModel + asyncpg (`ext.asyncio.session.AsyncSession`). DO NOT use pure SQLAlchemy models unless strictly necessary for pgvector operations.
- **Migrations**: Alembic (using Async mode, metadata mapped to SQLModel).

## Architecture & Data Flow Invariants
1. **Session Management (Client-Generated ID)**: 
   - The `session_id` (UUIDv4) is ALWAYS generated by the frontend.
   - Use `get_or_create` logic in the backend. Do not throw errors if a session already exists; treat it as an ongoing conversation.
2. **Context Assembly (Backend-Driven)**: 
   - The frontend only sends `{ session_id, message }` (the newest user message).
   - The backend is responsible for fetching the message history (sliding window, e.g., limit=30).
   - **CRITICAL**: When assembling the context payload for the LLM, the `System Prompt` MUST always be placed at `messages[0]`.
3. **Database Streaming Safety**:
   - The `AsyncSession` is bound to the HTTP request lifecycle via `Depends(get_session)`.
   - Never perform DB writes inside the `async for chunk in stream` loop. Yield all chunks to the frontend first, then commit the complete Assistant message to the database after `[DONE]` is sent, gracefully catching any DB disconnects without crashing the stream.
